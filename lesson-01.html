gRPC vs REST — The Why

Understand why gRPC exists, what problems it solves over REST, and why B2B control plane ↔ data plane architectures rely on it.

⏱ 30 min 📖 Concepts only 🎯 No code required

🌐 What is gRPC?

gRPC (Google Remote Procedure Call) is a high-performance, open-source framework for building APIs. Instead of sending JSON over HTTP/1.1 like REST, gRPC sends binary Protocol Buffer messages over HTTP/2.

Think of it like this: REST is a postcard (text, slow, one message at a time). gRPC is a dedicated phone line (binary, fast, bidirectional, always open).

🚀 HTTP/2 — What Makes gRPC Fast

gRPC is built on HTTP/2 which brings several fundamental improvements over HTTP/1.1:

🔗
Multiplexing
Multiple concurrent streams over a single TCP connection. No head-of-line blocking.
📦
Binary Framing
Data sent as binary frames, not text. Faster to parse, smaller on the wire.
🗜️
Header Compression
HPACK compression reduces header overhead by up to 85% on repeated calls.
⬆️⬇️
Server Push
Server can push data to client without a request. Enables streaming patterns.

📡 4 Types of gRPC Calls

One of gRPC's superpowers is native support for 4 communication patterns:

↔️
Unary
One request → one response. Like a REST call. Used for: RegisterClient, GetStatus.
📤
Server Streaming
One request → many responses. Server streams events. Used for: live logs, updates.
📥
Client Streaming
Many requests → one response. Client streams data. Used for: bulk uploads, metrics.
🔄
Bidirectional
Many requests ↔ many responses. Full duplex. Used for: persistent command streams between control plane and data plane.

🏗️ Why B2B Platforms Use gRPC

🔮
The Firewall Problem in B2B Deployments
In typical B2B SaaS platforms, gRPC solves a critical problem: data plane instances are deployed in customer networks behind corporate firewalls that block inbound connections. gRPC bidirectional streaming lets the data plane initiate an outbound connection to the control plane and then receive commands through that same persistent stream — no inbound ports needed on the customer side.

The key insight: the data plane opens the connection outbound. This means:

  • No inbound firewall rules needed on the customer's side — standard outbound HTTPS (443) is enough
  • The control plane can push commands to the data plane through the open stream
  • Heartbeats detect failures instantly and trigger reconnection
  • Multiple data plane instances can connect to one control plane simultaneously

🚫 When NOT to Use gRPC

⚠️
gRPC is not always the right choice
gRPC shines for internal microservices. But avoid it for public browser APIs (use REST + JSON), simple CRUD services where REST is simpler, teams unfamiliar with protobuf tooling, and when browser-native fetch() is required without a proxy.

💪 Knowledge Check

No code needed yet — test your understanding of the concepts.

Think about these questions:

1. Why does the data plane connect outbound to the control plane rather than the control plane connecting inbound to each data plane?

2. In the diagram, what happens if the gRPC stream between dplane and cplane drops? What mechanism detects this?

3. Why would a REST API be wrong for the cplane↔dplane connection even though it works fine for the platform UI?

4. What would happen to StreamMessages if it were implemented as a REST polling endpoint instead?

✅
Model Answers
1. Dplanes run in customer networks behind firewalls. Inbound ports to customer networks require firewall rules that customers often can't or won't open. Outbound HTTPS (443) is always open. gRPC bidirectional streaming over a single outbound connection solves this.

2. Heartbeat messages sent on a timer detect the drop. If no heartbeat ACK arrives within a timeout, the dplane knows the connection is dead and triggers reconnection with exponential backoff (Lesson 10).

3. REST would require polling (cplane can't push to dplane), each poll is a new HTTP connection (inefficient), and streaming commands would be impossible without WebSockets (complex, not HTTP/2 native).

4. Polling REST would miss messages, add latency proportional to poll interval, generate massive overhead (new TCP+HTTP overhead per poll), and make ordering guarantees very hard.
