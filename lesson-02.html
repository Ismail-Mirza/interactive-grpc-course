Proto3 Foundations

Learn the language of gRPC — Protocol Buffers. Write your first .proto file, understand types and field numbers, and generate Python stubs with protoc.

⏱ 45 min 🔨 First code 📁 Creates: cplane.proto

📄 What is a .proto File?

A .proto file is the single source of truth for your gRPC service. It defines:

  • The messages (data structures) that get sent over the wire
  • The services (RPC endpoints) that clients can call
  • The field types and numbers used for binary encoding

From one .proto file, protoc generates client and server code in Python, Go, Java, TypeScript, and 10+ other languages. This is what makes gRPC language-agnostic.

💡
Proto file is a contract
The .proto file is shared between both sides. In a real system, the control plane and data plane share the same cplane.proto as their contract. Change the proto on either side, regenerate both, and redeploy.

⚠️
Default values in proto3
In proto3, all fields have default values: string → "", int32 → 0, bool → False, bytes → b"". There is NO null — use optional keyword or oneof if you need to distinguish "not set" from the default value.

📦 Messages, Fields & Field Numbers

A message is proto3's equivalent of a class or struct. Each field has a type, a name, and a field number.

proto3 example.proto 
syntax = "proto3";

package example;

// A message is like a Python dataclass
message ClientInfo {
  string client_id   = 1;   // field number 1
  string tenant_id   = 2;   // field number 2
  string version     = 3;   // field number 3
  bool   is_active   = 4;   // field number 4
  int32  retry_count = 5;   // field number 5
}
🔢
Field numbers are CRITICAL — never change them
Field numbers (1, 2, 3...) are what get encoded in the binary wire format — NOT the field names. If you change a field number, old clients can't read new messages. Numbers 1–15 use 1 byte. Numbers 16–2047 use 2 bytes. Always use 1–15 for frequently-used fields.

📋 Enums, Repeated & Nested Messages

proto3 example.proto 
syntax = "proto3";

// Enums: named integer constants
enum MessageType {
  UNKNOWN         = 0;  // Always start at 0 in proto3
  CLIENT_REQUEST  = 1;
  SERVER_REQUEST  = 2;
  RESPONSE        = 3;
  HEARTBEAT       = 4;
}

// Repeated: like a Python list
message StreamPayload {
  repeated string tags = 1;   // ["tag1", "tag2"]
  repeated int32  ids  = 2;   // [1, 2, 3]
}

// Nested message: message inside message
message Wrapper {
  ClientInfo info    = 1;  // nested ClientInfo
  MessageType type   = 2;  // enum field
  StreamPayload data = 3;  // nested StreamPayload
}

🔌 Services and RPCs

A service defines the RPC methods that a gRPC server exposes. This is what becomes your server class and client stub.

proto3 example.proto 
syntax = "proto3";

// Request/Response messages
message RegisterRequest {
  string client_id  = 1;
  string tenant_id  = 2;
  string auth_token = 3;
}

message RegisterResponse {
  bool   success   = 1;
  string session_id = 2;
  string message   = 3;
}

// The service definition
service ControlPlane {
  // Unary RPC: one request, one response
  rpc RegisterClient (RegisterRequest) returns (RegisterResponse);

  // Server streaming: one request, many responses
  rpc ListConnectors (ListRequest) returns (stream ConnectorInfo);

  // Client streaming: many requests, one response
  rpc UploadLogs (stream LogEntry) returns (UploadResponse);

  // Bidirectional streaming: many ↔ many
  rpc StreamMessages (stream ClientMessage) returns (stream ServerMessage);
}
🔮
Real-world cplane.proto uses all 4 patterns
In a production B2B control plane, you'll find exactly these patterns: RegisterClient (unary), GetClientStatus (unary), ExecuteAPIRequest (unary), and StreamMessages (bidirectional streaming) — the most important one that keeps the data plane permanently connected.

⚙️ Generating Python Code with protoc

After writing your .proto file, you compile it with protoc to generate Python files:

bash terminal 
# Install tools
pip install grpcio grpcio-tools

# Generate Python stubs from proto file
python -m grpc_tools.protoc \
  -I proto/ \
  --python_out=proto/ \
  --grpc_python_out=proto/ \
  proto/cplane.proto

# This generates two files:
# proto/cplane_pb2.py       — message classes
# proto/cplane_pb2_grpc.py  — server/client stubs

What gets generated?

FileContainsYou use it for
cplane_pb2.pyMessage classes (RegisterRequest, RegisterResponse...)Creating/reading message objects
cplane_pb2_grpc.pyStub (client) + Servicer (server base class)Server: extend Servicer. Client: create Stub.
💡
Never edit generated files
The files ending in _pb2.py and _pb2_grpc.py are auto-generated. Never edit them manually — they get overwritten every time you run protoc. A common practice is to commit generated stubs to the repo so no compile step is needed at runtime on each deployment.

🛠️ Demo Project — Step 1: Your First Proto File

Create the foundation for the mini cplane↔dplane you'll build throughout this course:

bash terminal 
mkdir -p grpc-demo/proto grpc-demo/cplane grpc-demo/dplane
cd grpc-demo
pip install grpcio grpcio-tools
proto3 grpc-demo/proto/cplane.proto 
syntax = "proto3";

package cplane;

// ─── Enums ───────────────────────────────────────────
enum MessageType {
  UNKNOWN          = 0;
  CLIENT_REQUEST   = 1;
  SERVER_REQUEST   = 2;
  RESPONSE         = 3;
  HEARTBEAT        = 4;
}

enum ActionType {
  GET    = 0;
  POST   = 1;
  PUT    = 2;
  DELETE = 3;
  PATCH  = 4;
}

// ─── Registration ────────────────────────────────────
message RegisterRequest {
  string client_id   = 1;
  string tenant_id   = 2;
  string auth_token  = 3;
  string version     = 4;
}

message RegisterResponse {
  bool   success    = 1;
  string session_id = 2;
  string message    = 3;
}

// ─── Streaming Messages ──────────────────────────────
message ClientMessage {
  string      message_id = 1;
  MessageType type       = 2;
  string      payload    = 3;
  string      client_id  = 4;
}

message ServerMessage {
  string      message_id = 1;
  MessageType type       = 2;
  string      payload    = 3;
  ActionType  action     = 4;
}

// ─── Service ─────────────────────────────────────────
service ControlPlane {
  rpc RegisterClient  (RegisterRequest)          returns (RegisterResponse);
  rpc StreamMessages  (stream ClientMessage)     returns (stream ServerMessage);
}

💪 Exercise

Extend the proto file with new messages.

Add to cplane.proto:

1. Add a ClientStatus enum with values: CONNECTED, DISCONNECTED, ERROR

2. Add a GetStatusRequest message with fields: client_id (string), session_id (string)

3. Add a GetStatusResponse message with fields: client_id, status (ClientStatus enum), last_heartbeat (int64 timestamp)

4. Add a GetClientStatus unary RPC to the ControlPlane service

proto3Solution
// Add this enum
enum ClientStatus {
  CONNECTED    = 0;
  DISCONNECTED = 1;
  ERROR        = 2;
}

// Add these messages
message GetStatusRequest {
  string client_id  = 1;
  string session_id = 2;
}

message GetStatusResponse {
  string       client_id      = 1;
  ClientStatus status         = 2;
  int64        last_heartbeat = 3;
}

// Update the service
service ControlPlane {
  rpc RegisterClient  (RegisterRequest)      returns (RegisterResponse);
  rpc GetClientStatus (GetStatusRequest)     returns (GetStatusResponse);
  rpc StreamMessages  (stream ClientMessage) returns (stream ServerMessage);
}
